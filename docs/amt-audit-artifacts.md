# AMT DOM-Signature Audit — Artefact Layout and Lifecycle

> **Purpose**: define where AMT DOM-signature audit output lives, how it
> moves between machines, and what each file represents. Covers the
> output of `scripts/audit-operator.ts` (Task A.5) and the
> upload/download pipeline that ships results from EC2 to local dev.
>
> **Companion**: `docs/amt-operator-spec.md` (the operators being
> audited), `.kiro/steering/project-context.md` §Experiment Data
> Pipeline (the equivalent for experiment trace data).

## 1. Directory structure per run

Each audit invocation produces a self-contained directory called a
**run-dir**. The layout is flat on purpose — easy to tar, easy to diff,
easy to open in a file manager when inspecting misalignment findings.

```
data/amt-audit-runs/<run-id>/
├── audit.json               Per-operator 12-dim signatures + run metadata
├── run.log                  stderr diagnostics captured during the run
└── screenshots/
    ├── H1_before.png
    ├── H1_after.png
    ├── ...
    ├── L13_before.png
    └── L13_after.png
```

### 1.1 Run ID

Format: `amt-audit-<unix-epoch-ms>` (e.g. `amt-audit-1777360153991`).
Auto-generated by the CLI when `--run-dir` is not passed explicitly.
The timestamp is the only disambiguator between concurrent runs on the
same machine; we do not include URL, operator set, or app because those
are inside the manifest.

### 1.2 `audit.json`

Shape (schema v1):

```jsonc
{
  "schemaVersion": "amt-dom-signature/v1",
  "timestamp": "2026-04-28T14:02:26.968Z",
  "fixture": {
    "url": "http://10.0.1.50:7770/",
    "loginApp": "shopping",
    "loginBaseUrl": "http://10.0.1.50:7770",
    "authenticated": true
  },
  "settleMs": 1500,
  "operators": {
    "L1": {
      "D1_tagCountDeltas": { ... },
      "D2_attrChanges": { ... },
      "D3_nodeCountDelta": 0,
      "A1_rolesChanged": 12,
      "A2_namesChanged": 12,
      "A3_ariaStateChanges": {},
      "V1_ssim": 0.82,
      "V2_maxBBoxShift_px": 219,
      "V3_meanContrastDelta": 0,
      "F1_interactiveCountDelta": 0,
      "F2_inlineHandlerDelta": 0,
      "F3_focusableCountDelta": 0,
      "changesReturned": 8,
      "durationMs": 2019,
      "screenshots": {
        "before": "screenshots/L1_before.png",
        "after":  "screenshots/L1_after.png"
      }
    },
    ...
  },
  "errors": { ... },
  "totalDurationMs": 32915
}
```

Screenshot paths are **relative to the run-dir**. Any downstream tool
reading `audit.json` should resolve paths via `runDir + "/" + path`.
This lets you tar a run-dir, move it to another machine, and still have
a working reference graph.

### 1.3 `run.log`

Verbatim copy of the stderr stream, including login diagnostics,
per-operator timing, and any warnings. Treat as forensic evidence —
useful when a signature reads zero and you want to know whether the
operator ran but hit an empty target set, or the injection itself
errored.

### 1.4 `screenshots/`

Two PNGs per operator, `before` and `after`. 1280×720 viewport,
`fullPage: false`. Used by SSIM / bbox / contrast computations;
retained for paper gallery and human misalignment review.

## 2. Lifecycle

Three phases, matching the existing experiment pipeline.

```
┌────────────┐    tar+upload     ┌────────────────┐   download+extract  ┌────────────────┐
│  EC2       │ ─────────────────▶│  S3            │ ──────────────────▶│  Local dev     │
│            │                   │                │                     │                │
│ Platform   │ audit-upload.sh   │ audits/        │  audit-download.sh  │ data/amt-audit │
│ runs audit │                   │ <runid>.tar.gz │                     │ -runs/<runid>/ │
│            │                   │ <runid>-       │                     │                │
│ data/amt-  │                   │   manifest.txt │                     │                │
│ audit-runs │                   │                │                     │                │
└────────────┘                   └────────────────┘                     └────────────────┘
```

### 2.1 On EC2 (production runner)

```bash
# One-off audit
npx tsx scripts/audit-operator.ts \
    --url http://10.0.1.50:7770/ \
    --login-app shopping \
    --output-dir data/amt-audit-runs

# After completion (manual):
bash scripts/audit-upload.sh amt-audit-1777360153991
```

### 2.2 S3 layout

```
s3://<bucket>/audits/
├── amt-audit-1777360153991.tar.gz           the run-dir archived
├── amt-audit-1777360153991-manifest.txt     file listing with sizes
└── ...
```

Separate prefix `audits/` (not `experiments/`) so the existing
`experiment-download.sh --list` does not see audit runs, and vice
versa. Audits are a different data lifecycle: small, regenerable,
per-URL rather than per-task-trace.

### 2.3 On local

```bash
# List all audit runs in S3
bash scripts/audit-download.sh --list

# Pull one
bash scripts/audit-download.sh amt-audit-1777360153991

# Pull the latest
bash scripts/audit-download.sh --latest
```

Download extracts to `data/amt-audit-runs/<run-id>/`, mirroring the
EC2-side path. Screenshot relative paths inside `audit.json` resolve
correctly on both sides.

## 3. What is NOT kept

- **`tmpdir()` intermediate files**: none. Screenshots now write
  directly to the run-dir. No orphan PNGs in `/tmp` or
  `/var/folders/...` after a run.
- **`results/amt/dom-signatures/*.json`**: the previous default output
  location (tracked only by `.gitignore`). The CLI still accepts
  `--output <path>` for single-file JSON output into that location, but
  this is a legacy compatibility mode that does NOT capture screenshots
  or run logs. New callers should use `--output-dir` (the default).
- **Absolute local paths in `audit.json`**: removed. `screenshots.*`
  are relative to the run-dir. The `fixture.url` field may contain an
  absolute `file://` URL when auditing a local fixture — this is
  acceptable for local debugging but such runs should not be
  uploaded to S3 without stripping that field.

## 4. `.gitignore` coverage

All audit artefacts are covered by the repo-wide `data/` ignore (line
3 of `.gitignore`). No per-audit rules needed.

`results/amt/` remains ignored (for historic legacy output paths).

## 5. CLI contract

```
npx tsx scripts/audit-operator.ts \
  --url <url>                            required
  [--operators L1,H2,...]                default: all 26
  [--login-app shopping|...]             default: anonymous
  [--login-base-url URL]                 default: derived from --url
  [--output-dir <path>]                  default: data/amt-audit-runs/
  [--run-id <id>]                        default: amt-audit-<unix-ms>
  [--settle-ms 1500]                     default: 1500
  [--output <legacy-json-path>]          legacy single-file JSON output
                                         (no screenshots/log saved)
```

Producing a run-dir is the default. Legacy single-file output (via
`--output`) is retained for back-compat with smoke scripts already
committed, but emits a warning and skips the screenshots + log
side-outputs.

## 6. Troubleshooting

### "S3 bucket not found" during audit-upload.sh

Same root cause as experiment-upload: the script discovers the bucket
by name prefix `a11y-platform-data-*`. Override via `S3_BUCKET` env
var if you're on a weird account setup.

### audit.json has screenshots.before but the file doesn't exist

Check `run.log` for that operator — likely a Playwright navigation
error before the BEFORE screenshot was written. The signature row
will still exist (with nulls) so you can see WHICH operator failed.

### I want to re-run the audit on an already-uploaded run

Nothing special — just run the audit again. Each invocation creates a
new run-id. S3 keeps all historic runs; download them by name.
